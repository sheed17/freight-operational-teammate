# Adapter Family 13 — Notification Delivery Adapter *(A18)*

*Registry + defaults apply.*

**Purpose.** Deliver outbound operational notifications (status updates, reminders, alerts) to parties — as **gated `SEND_OUTBOUND` effects** where they carry commitments, or informational sends. **Not.** ### **NOT the oversight surface (that is A14 — it transports human DECISIONS); NOT authoritative for delivery/receipt.** **Direction.** outbound. **Vendors.** email (via A1), SMS (via A2), push — same contract.

**Ops.** `send_notification` (### **`CONSEQUENTIAL_EFFECT`/`SEND_OUTBOUND` when it makes a commitment to a counterparty — e.g. "your appointment is confirmed for 14:00"; else informational**), verification = ### **RECEIPT_VERIFIABLE (relay/carrier receipt proves TRANSMISSION, not delivery/receipt/read — M-72); a bounce ⇒ an Observation ⇒ retry/Expectation.** **Idempotency.** ### **a notification carries a `commit_key`/dedup identity so a retry does not double-send (a repeated "confirmed" is one notification).** **Unknown-outcome.** ### **accepted-then-unknown-delivery (H28) ⇒ receipt proves transmission; delivery `unknown`; NEVER recorded as "delivered".** **Security.** outbound content is generated by Neyma from canonical facts — ### **it never echoes unsanitized inbound content that could carry an injection to a counterparty.** **Acceptance.** `test_a18_send_receipt_is_not_delivery` (H28); `test_a18_notification_idempotent_no_double_send`; `test_a18_commitment_notification_is_gated`. **Adversarial.** H28. **Open.** which notifications carry commitments (per workflow) — NEEDS VALIDATION.
