# Domain Family 11 — Integration & Projection *(External Entity Mapping · Operational Timeline Entry · Financial Reconciliation Result)*

*Registry applies.*

## E38 — External Entity Mapping
**Def.** ### **The native record that binds a Neyma domain entity to its representation(s) in an external system (a TMS row, a portal record) — the ONLY way a TMS "load record" enters the model.** **Not.** ### **A domain entity itself** (there is no "TMS Load" entity); not an Identity Binding Claim about documents (this maps *entities to external ids*). **Authority.** native (the mapping is Neyma's); the external record's *content* is projected via Observations. **Identifier.** `mapping_id`. **Attrs.** `neyma_entity_type`+`neyma_entity_id`, `external_system`, `external_id`, `provenance_class`, `confidence` (ranks only). **Cardinality.** ### **Neyma entity 1:N external records (one Load in several TMSs); external record ... :N Neyma entities — hostile #26: ONE TMS row holds TWO business entities ⇒ the mapping is per-`(external_system, external_id, field-scope)`, so two entities map into one row without collapsing (CD-19).** **Identity.** ### **the external id is trusted ONLY within `(tenant, external_system)`; a `load_ref` across TMSs or tenants is a candidate signal, never a confirmation (GR-8).** **CD-6/CD-7.** **Hostiles:** #25 (two TMS records → one Load) ⇒ 1:N mapping; #30 (an event arrives for an entity whose mapping is not yet created) ⇒ ### **parked in `pending_references`, drained on mapping creation, TTL ⇒ Exception (M-26).** **Acceptance.** `test_no_tms_load_entity_exists`; `test_external_id_trusted_only_within_system_and_tenant`; `test_two_entities_one_tms_row_no_collapse`; `test_event_before_mapping_is_parked`. **Invalid-impossible.** a domain "TMS Load" entity; a cross-tenant external-id match. **Open.** per-TMS id semantics — NEEDS VALIDATION.

## E39 — Operational Timeline Entry
**Def.** ### **A derived, read-only VIEW row — a human-readable "what happened, when, why" entry** assembled from Audit/Business events for a Load/Movement/party. **Not.** ### **A source of truth** (it is derived; ### **rebuildable from events — never authoritative**); not an Audit Event (it is a projection *of* them). **Authority.** ### **native derived (a materialized projection) — updates ONLY from events (M-2); never optimistically.** **Identifier.** `timeline_entry_id`. **Rebuild.** ### **fully rebuildable from the event corpus; a rebuild divergence ⇒ `ProjectionRebuildDiverged` ⇒ auto-brake.** **Provenance.** each entry pins the events it derived from (evidence traversal, M-12). **Use.** the "why did you do that?" explainability surface; ### **explains using the beliefs OF THAT DAY (M-68), not today's.** **Acceptance.** `test_timeline_is_derived_rebuildable_not_authoritative`; `test_timeline_explains_with_beliefs_of_that_day`. **Invalid-impossible.** a consequential action reading the timeline as authority. **Open.** none.

## E40 — Financial Reconciliation Result · lifecycle **L-Recon**
**Def.** ### **A derived result comparing expected vs actual financials for a Load/Movement** — margin realized, AR vs invoiced, AP vs paid, accessorial capture, discrepancies. **Not.** ### **An Invoice/Payable/Payment** (those are the sources); not authoritative money (it is a computed comparison). **Authority.** native derived. **Identifier.** `reconciliation_id`. **Attrs.** `expected` vs `actual` per money_kind (IN/OUT explicit), `discrepancies[]`, `status`. **L-Recon states.** `COMPUTED → RECONCILED` · `DISCREPANT` · `SUPERSEDED`. **Key transitions:** `DISCREPANT` (a mismatch — e.g. #29 late Rate Con vs completed payable, #19 short-pay, #6 rate-con-vs-phone) ⇒ ### **`ConflictRaised` + a Work Item; NEVER a silent adjustment (CD-4/CD-16).** **Provenance.** pins every source financial record (CD-18 evidence chain). **CD-18.** **Money.** ### **every value carries direction + kind (CD-17); expected margin = expected sell − expected buy, computed.** **Acceptance.** `test_recon_is_derived_not_authoritative_money`; `test_recon_discrepancy_raises_conflict_not_silent_adjust`; `test_recon_preserves_full_evidence_chain`. **Invalid-impossible.** a reconciliation silently adjusting an Invoice/Payable. **Open.** discrepancy thresholds — NEEDS VALIDATION (fail-closed to a Conflict/human).
