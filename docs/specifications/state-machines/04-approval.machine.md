# Machine M4 — Approval

*Registry: `registry.md`. Entity: `entities/06-approval.md`. Lifecycle: Target Spec §12.4; ADR-005.*

**1.** Approval. **2.** Approval. **3. Purpose.** An authenticated human agreeing to an action **and its exact material facts**; authorizes **one committed effect, not one network attempt**. **4. Aggregate/txn.** Own aggregate; ### **`GRANTED→CONSUMED` co-commits with the M3 claim CAS.** **5. Owner.** the approving human. **6. Initial.** `REQUESTED`. **7. States.** the 8 (registry §4/M4). **8. Terminal.** `CONSUMED, DENIED, EXPIRED, REVOKED, VOID_ON_DRIFT, VOID_ON_BRAKE`. **9. Non-terminal.** `REQUESTED, GRANTED`. **10. Recoverable.** `REQUESTED, GRANTED`. **11. Failure.** none (a denied/void approval is terminal, not a failure). **12. Expiry.** `EXPIRED` (TTL). **13. Unknown-outcome.** ### **none — but a `GRANTED` approval whose attempt went `UNKNOWN_OUTCOME` is FROZEN (not reusable) until reality is established.**

**Dual control:** `AP-2` requires **quorum by distinct authenticated actors**, each `ApprovalSignature` binding the **same fingerprint**; ### **drift between signatures voids ALL signatures** ⇒ back to `REQUESTED` with a fresh fingerprint.

## 15. Illegal transitions. `CONSUMED`+anything → ILLEGAL (single-use). A partial approval ("approve but £2,700") → not a transition; a **new** `REQUESTED` with a new fingerprint. Grant by a model/counterparty → ILLEGAL (Sev-0 fraud signal). Reuse of a frozen (AP-9) approval → ILLEGAL. **16. Precedence.** drift/policy/brake voids (AP-4/4p/5) > consume (AP-7): a void racing a claim ⇒ OCC/CAS admits one; if the void commits first, the claim's CAS fails. **17. Concurrency.** the consume-CAS co-serialized with the M3 claim CAS. **18. Version-check.** the fingerprint IS the material-fact+version check (recomputed live at PL-8 step 2). **19. Idempotency.** ### **a double-tap of the Slack button finds `CONSUMED` and replies "already done", raising nothing, acting nothing.** **20. Retry.** an approval survives a provable failure (AP-8) and may authorize a **new** pipeline instance under the **same commit_key**; consumed exactly once. **21. Replay.** GR-11 — never re-grants, never consumes into an effect.
## 22. Cancellation. `REVOKED` (AP-6) before consume. **23. Expiry.** AP-3. **24. Reopening.** n/a. **25. Correction.** a changed decision is a new approval. **26. Supersession.** ### **no `SUPERSEDED` state — supersession = drift-void ∪ duplicate-refusal.** **27. Compensation.** n/a. **28. Human-approval.** (self). **29. Policy.** AP-4p. **30. Brake.** AP-5.
## 31. Evidence/provenance. `provenance_class` of every material field is **inside** the fingerprint (M-56); `granted_by` is `OWNER_ASSERTED`-grade. **32. Events.** AP-* per registry. **33. Consumes.** `HumanApproved, HumanDenied, MaterialFactsChanged, PolicyVersionChanged, BrakeEngaged, HumanRevoked, EffectCommitted, AttemptFailedProvably, AttemptOutcomeUnknown, TimerFired`. **34. Writes.** state + full `canonical_payload` (retained) + outbox. **35. Txn.** the consume-CAS co-commit. **36. Crash recovery.** a `GRANTED` approval survives a crash; recovery re-runs the checkpoint (incl. live drift) before any claim; a crash after `CONSUMED`-before-effect ⇒ M3 `UNKNOWN_OUTCOME`, approval **not** returned to `GRANTED`. **37. Timeout.** AP-3 (TTL). **38. Observability.** drift-void rate + time-to-approve are first-class; every void explainable in one message (the `drift_diff`). **39. Audit.** every request/grant/void/expiry/consume with actor; the retained payload is the evidentiary heart. **40. Security.** created only by an authenticated authorized human; counterparty claims are `MODEL_EXTRACTED` fraud signals (ADR-003); token single-use + actor-bound.
