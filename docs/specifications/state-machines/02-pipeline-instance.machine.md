# Machine M2 — Pipeline Instance

*Registry: `registry.md`. Entity: `entities/02-pipeline-instance.md`. Lifecycle: Target Spec §12.2, §19.*

**1.** Pipeline Instance. **2.** Pipeline Instance. **3. Purpose.** One durable attempt to produce one effect; the Pipeline Instance IS the command. **4. Aggregate/txn.** Own aggregate; **the `CHECKPOINT→GRANTED` transition co-commits the Checkpoint Witness insert + the grant mint (M3) in ONE transaction**; the `GRANTED→CLAIMED` transition co-commits with M3's claim CAS + M4 `CONSUMED`; all others one commit. **5. Owner.** the Work Item's owner. **6. Initial.** `PROPOSED`. **7. States.** the 16 of M2 (registry §4). **8. Terminal.** `CLOSED, REJECTED, VOIDED, FAILED`. **9. Non-terminal human-owned.** `AWAITING_APPROVAL, NEEDS_VERIFICATION`. **10. Recoverable.** `PROPOSED, POLICY_CHECKED, VALIDATED, CHECKPOINT, GRANTED, CLAIMED, EXECUTED, VERIFIED, RECORDED, PROJECTED`. **11. Failure.** `FAILED` (provable non-occurrence only). **12. Expiry.** grant/approval TTL ⇒ `VOIDED` (pre-claim). **13. Unknown-outcome.** `NEEDS_VERIFICATION`.

**Machine defaults:** consequential transitions = **PL-8 (checkpoint) and PL-9 (claim) only**; all others administrative. Owner-after = the Work Item's owner throughout.

## M2↔M3 co-transition rule
> ### **`PL-8` (mint) and `PL-9` (claim) are co-transitions with M3.** The pipeline's `GRANTED`/`CLAIMED` **reflect** the single `effect_grants` row's `GRANTED`/`CLAIMED` state (SD-2). M2 does not own the grant state — **M3 does** — but the two rows transition in the same commit. Neither may be in `CLAIMED` without the other.

## 15. Illegal transitions *(GR-1)*
`GRANTED` reachable without a `CheckpointPassed` witness → impossible (PL-8 co-commits). `CLAIMED` without a successful CAS → impossible. Any timer on `NEEDS_VERIFICATION` → ILLEGAL. Retry-in-place (mutating a terminal instance) → ILLEGAL (a retry is a **new** instance). `CLAIMED`→`FAILED` without proof → ILLEGAL. Skipping the checkpoint (`VALIDATED`/`AWAITING_APPROVAL`→`GRANTED`) → ILLEGAL.

## 16. Precedence. `BrakeEngaged` and `PolicyVersionChanged` **void** a pre-claim pipeline (PL-7v/PL-9v) with priority over advancement. Drift (PL-7v via M4 `VOID_ON_DRIFT`) > approval-bind. **17. Concurrency.** OCC + **the claim CAS is the serialization point** — two attempts to claim one grant: exactly one wins (GR-3, single-use). **18. Version-check.** GR-13 (the SD-3 entity-version set) at PL-8, re-validated at PL-9 CAS. **19. Idempotency.** GR-4; the claim CAS makes PL-9 idempotent (a redelivered claim finds the grant not-`GRANTED`, no-op). **20. Retry.** ### **NEVER in place — a NEW Pipeline Instance, same `commit_key`, new grant, full checkpoint (incl. drift).** Only after a **provable** `FAILED` (PL-10f) or a pre-claim `VOIDED`; `NEEDS_VERIFICATION` blocks any retry (Layer-1 held). **21. Replay.** GR-11 — replaying M2 mints zero grants, produces zero `EffectAttempted`.
## 22. Cancellation. Only **before `CLAIMED`** (⇒ `VOIDED`, PL-7v/PL-9v). After `CLAIMED`, undo = M10 Compensation. **23. Expiry.** grant/approval TTL pre-claim ⇒ `VOIDED`; `NEEDS_VERIFICATION` never expires. **24. Reopening.** never (reopening is at M1). **25. Correction.** a wrong proposal is `REJECTED`/`VOIDED`; a corrected attempt is a new instance. **26. Supersession.** n/a. **27. Compensation.** a wrong verified effect ⇒ M10 (a separate gated pipeline).
## 28. Human-approval. PL-6/PL-7b bind M4 to **this** commit_key + fingerprint; drift ⇒ M4 `VOID_ON_DRIFT` ⇒ PL-7v. **An approval authorizes one committed effect (one claim), not one network attempt.** **29. Policy.** step 6 at PL-8; a policy change pre-claim ⇒ PL-7v/PL-9v void. **30. Brake.** step 7 at PL-8 + re-validation at PL-9 CAS; ### **a brake engaged between PL-8 and PL-9 makes the CAS match zero rows — the adapter does nothing (never both, never neither).** A post-claim effect proceeds to verification (GR-16).
## 31. Evidence/provenance. GR-8 (no `MODEL_INFERRED` material fact), GR-10 (no open Conflict on a material field), GR-13. **32. Events emitted.** PL-* + co-emitted M3/M4 events per registry §5. **33. Events consumed.** `IntentProposed, ApprovalGranted/Denied/Expired, PolicyVersionChanged, BrakeEngaged, GrantExpired/Revoked, AdapterReturnedSuccess/RejectedPreFlight/TimedOut, Readback*, VerificationDeferred, HumanEstablishedReality, LaterObservationProves`. **34. Durable writes.** state row + witness + grant + approval-consume + outbox, in the co-commit txns above. **35. Txn boundaries.** the two co-commit transactions (PL-8, PL-9) are the safety-critical ones; **no async work between PL-8 and PL-9**. **36. Crash recovery.** pre-effect stage ⇒ re-run the checkpoint (nothing happened); `CLAIMED`/`EXECUTED` ⇒ ### **UNKNOWN OUTCOME, never re-execute** (PL-10u); recovery re-derives or escalates, never guesses. **37. Timeout.** grant/approval TTL (void); deferred-verification recheck (PL-11d); **no timeout moves `NEEDS_VERIFICATION`.** **38. Observability.** `NEEDS_VERIFICATION` is a Sev-1 queue with owner/age/exposure/the-specific-question; `CheckpointFailed{step}` distribution monitored. **39. Audit.** every transition; PL-8/PL-9/PL-8f are load-bearing for explainability. **40. Security.** the checkpoint's type-level witness + the CAS + the import-graph gate make bypass impossible (GR-11, §19.9); agents emit inert proposals only.

## 41. Acceptance criteria.
(a) `GRANTED` unreachable without a witness; (b) drift after approval voids (F-01); (c) a retry is a new instance with the same commit_key; (d) `NEEDS_VERIFICATION` never auto-resolves; (e) the brake-vs-claim race is never-both-never-neither; (f) verify+record is one commit.
## 42. Adversarial scenarios *(traced in the cross-review)*.
Trace 1 (drift), 2 (brake between checkpoint and claim), 3 (policy change pending), 4 (two pipelines one effect), 9 (verification unavailable), 19 (entity version changed after approval), 20 (grant claimed twice) all resolve on this machine — see `state-machine-specification-review.md`.
## 43. Open validation. **V6** (deferred-verification bounds per TMS). Fail-closed default: treat as `AWAITING_OBSERVATION` + Expectation. Not a block.
