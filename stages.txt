Stage 1 — Extraction proof (no Browser Use, no Slack, no TMS, no agent)
This is where you actually start, and it's simpler than it sounds. You're proving one thing: can the vision model read a real carrier invoice and return the right structured fields with confidence scores. That's it. The full stack for this stage is: Python, PyMuPDF, one vision LLM API call, Pydantic, Instructor. No Browser Use. No Slack. No database. No state machine. Not even a full pipeline yet — just a script that takes a PDF, renders it, calls the model, returns a structured object. You test it against your golden set, score it field by field, iterate the prompt until the accuracy is solid.
You'll know this stage is done when: extraction on 20 real carrier invoices scores accurately enough on the required fields (load/PRO number, linehaul, fuel surcharge, total) that you trust the data it produces. That's the gate. Do not move on until this is solid, because everything downstream depends on it.
Stage 2 — Config system + reconciliation matching (still no Browser Use, no Slack, no TMS)
Now you make the extraction config-driven (fields/prompt/rules come from YAML, not hardcoded), and you build the matching logic — the deterministic Python that takes the extracted invoice fields and compares them against a rate confirmation. At this stage the rate confirmation is a fixture (a hardcoded test object, not pulled from a real TMS) — you're proving the matching logic works, not the TMS connection. Test it against known pairs: some that match cleanly, some with deliberate variances (wrong linehaul, extra accessorial, duplicate). Confirm it flags the right ones with the right reasons.
You'll know this stage is done when: the config system is working, and the matching logic correctly identifies matches and variances on your test pairs.
Stage 3 — State machine + orchestrator + database (still no Browser Use, no Slack, no TMS)
Now you build the spine: the SQLite database, the state machine (the document lifecycle states), and the polling loop that moves documents through stages. This is purely deterministic Python — no LLM involved here. You're building the conveyor belt that connects extraction → matching → review → entry. At this point entry is a stub (logs "would enter: {data}" and marks the document ENTERED). Test it by running a document through the full loop: ingest → extract → match → classify into a bucket → advance to PENDING_REVIEW → stub-enter → DONE. Confirm idempotency (same document twice doesn't double-process).
Stage 4 — Slack HITL gateway (still no Browser Use, no real TMS)
Now you wire the human-in-the-loop layer. FastAPI + slack-sdk. Documents that need review get pushed to Slack as Block Kit cards showing the extracted fields, the variance detail, and the Approve/Dispute buttons. When she clicks Approve in Slack, the webhook fires, the chassis moves the document to APPROVED. At this stage entry is still a stub. Test it end-to-end: a real carrier invoice runs through the pipeline, a Slack card appears with the right fields and variance, you click Approve, the document advances. Verify the Slack request signatures are being checked.
You'll know this stage is done when: a real document flows through, surfaces in Slack correctly, and a button click advances the state. The human gate is working.
Stage 5 — Browser Use agent for TMS READ (first time Browser Use appears)
Now Browser Use enters — and notice it's stage 5, not stage 1. The first Browser Use job is the READ operation: pull the rate confirmation from the client's TMS by load/PRO number, so you have real reference data for matching instead of fixtures. You build this against a mock TMS first (a local dummy page that looks like a TMS search form) to prove the mechanics. Then you wire it to a real TMS in their environment (sandbox/trial account, never their live data).
This is when the session-persistence and no-credentials rules matter for real: Browser Use mounts the persisted session profile, navigates to the load search, returns the rate con fields. If there's no valid session, it returns NO_SESSION and the chassis routes to WAITING_FOR_SESSION and Slacks the human to re-login.
You'll know this stage is done when: Browser Use can reliably pull rate con data from the mock TMS (and then from the real one in a test environment).
Stage 6 — Browser Use agent for TMS WRITE (the entry action)
The second Browser Use job is the WRITE operation: on an APPROVED document, navigate to the right TMS screen, fill the fields with the approved data, submit, verify by reading back. This is the highest-stakes stage — it's touching the client's real system — so you build it last, mock-first, with the verify-don't-trust rule enforced. The chassis only marks ENTERED after independent verification.
Test it extensively on the mock form, then on a TMS sandbox/trial account, before ever pointing it at a client's live system. First real entries are supervised (you're watching, she's approving each one, you're verifying what was entered).
Stage 7 — Full loop test + deployment
Run the complete pipeline end-to-end on real documents from a real client in a real environment: ingest from email → extract → match against real TMS rate con (Browser Use READ) → classify → Slack card → human approves → Browser Use WRITE → verify → DONE. Test every unhappy path: variance found, no rate con match, low confidence, duplicate invoice, session expired, entry failed. Confirm each lands in the right safe state with a clear Slack notification.
Then deploy: package in Docker, provision the host (VM if TMS is web-reachable, office machine if not), set up the email forwarding trigger, establish the client's TMS session, set the schedule. First week is fully supervised — she approves everything, you watch every entry, you verify each one.
Stage 8 — Harden against real failures
After a week of real use, you'll see the actual failure modes — maybe one carrier's invoices have an inconsistent format, maybe the TMS occasionally loads slowly and Browser Use times out, maybe a specific accessorial code isn't being extracted correctly. You fix those specific real failures (not theoretical ones), tighten the prompts, adjust confidence thresholds based on real accuracy data. This is v1.5 — hardening against what actually breaks, not what you predicted would break.
When does Browser Use actually come in? Stage 5. Everything before it is pure Python + LLM + Slack. You don't need Browser Use until you're pulling rate con data from a real TMS, which is stage 5. A common mistake is trying to build the browser automation early — it's the hardest, most environment-dependent part, and building it before the extraction and matching are proven means debugging two hard things at once.
